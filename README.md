# GMap — India Road Network & Algorithm Visualizer

> A production-grade graph routing engine built in C++ with a Google Maps-style frontend.
> Implements Dijkstra, A*, and Bidirectional Dijkstra with live benchmarking on real OSM data.

---

## Live Demo

Open `GMap_v3.html` in any browser — no server, no installation required.
All algorithms run in JavaScript mirroring the C++ implementation.

---

## What This Project Does

A complete map routing system that solves two problems:

**1. Shortest Path** — Given two Indian cities, find the fastest road route.
**2. Optimal Road Trip** — Given N cities, find the most efficient order to visit all of them.

Both problems are solved with multiple algorithms so their efficiency can be compared directly.

## Algorithms

### 1. Dijkstra's Shortest Path

**Time:** O((V + E) log V) | **Space:** O(V)

Explores cities in order of distance from the source using a min-heap. Guaranteed optimal on non-negative weighted graphs.

**Why adjacency list?** The graph is sparse (avg ~3.5 edges per city). An adjacency matrix for 268 cities wastes 71,824 cells, 99.4% empty. The adjacency list uses exactly O(V + E) entries.

### 2. A* (A-Star)

**Time:** O((V + E) log V) | **Practical:** 45-50% fewer nodes than Dijkstra

Adds a heuristic h(n) to guide the search toward the destination:

```
f(n) = g(n) + h(n)
         |       |
   actual road   Haversine straight-line
   dist so far   dist to destination
```

**Why Haversine?** India spans 30 degrees latitude. Flat Euclidean distance is inaccurate at this scale. Haversine computes the exact great-circle distance and never overestimates (admissible) so A* stays optimal.

**Heuristic quality measured:** 52% average accuracy on Indian roads. Roads are moderately winding due to mountains and rivers.

### 3. Bidirectional Dijkstra

**Time:** O((V + E) log V) | **Practical:** ~21% fewer nodes

Two simultaneous searches from both ends meet in the middle. Each covers radius d/2, total area = half of one-way search.

### 4. Road Trip Planner (TSP)

TSP is NP-hard. Two heuristics combined:
- **Nearest Neighbour** — greedy construction, O(n2), within 20-25% of optimal
- **2-opt** — swap edges to eliminate crossings, cuts another 5-15%

## Bottleneck Analysis

Phase-level profiling on 268-city graph, 300 queries:

**Dijkstra — 18.3 µs per query**
```
Heap operations    47%  ← primary bottleneck
Edge relaxation    51%  ← primary bottleneck
Initialisation      1%
Path reconstruct    1%
```

**A* — 26.2 µs per query**
```
Edge relaxation    41%
Heuristic cost     39%  ← cost of guidance
Heap operations    19%  (fewer because fewer nodes visited)
```

**Key finding:** Priority queue accounts for 47% of Dijkstra time. This is the #1 optimisation target.

**Production optimisations:**
1. Contraction Hierarchies — ~1000x speedup (offline preprocessing)
2. Fibonacci Heap — 2-3x speedup (O(1) decrease-key)
3. Heuristic caching — 10-20% (avoid recomputing h per neighbour)
4. Pre-allocated arrays — 5-10% (avoid malloc per query)

## Why Not Google Maps API?

This project implements the routing engine itself, not a wrapper. The goal is to understand and demonstrate the algorithms that power navigation — the same algorithms (Dijkstra, A*, Bidirectional) that Google Maps, Apple Maps, and OSRM all use internally.

---

## How Google Maps Works

Google Maps uses Contraction Hierarchies (CH) on top of A*:

**Offline (done once, takes days):**
- Rank every node by importance (shortest paths through it)
- Add shortcut edges that skip unimportant nodes

**Online (milliseconds):**
- Run bidirectional A* on high-importance nodes only
- Result: 1 billion nodes, 10ms query time

This project implements the A* + Bidirectional foundation. CH preprocessing is the remaining gap.

## Design Decisions

**Why C++?** Direct memory control, no GC pauses, same language as production routing engines at Google, HERE, TomTom.

**Why adjacency list?** Sparse graph. Matrix wastes O(V2) space, 99% empty.

**Why priority_queue over sorted array?** O(log V) insert vs O(V) for sorted array. Priority queue wins at our query sizes.

**Why Haversine for A*?** Earth curvature matters at India scale. Haversine is exact great-circle distance, always admissible.

**Why NN + 2-opt for TSP?** Exact TSP is O(n!). For 20 cities = 2.4 quintillion operations. NN + 2-opt gives within 5-15% of optimal in O(n2).
